    Add `discoverModelsForKey()` to `src/lib/server/provider-keys.ts` with this exact signature and implementation:

    ```typescript
    export function discoverModelsForKey(
      keyId: string,
      provider: string,
      apiKey: string,
      baseUrl?: string
    ): void {
      validateProviderKey(provider, apiKey, baseUrl)
        .then(async (result) => {
          if (result.valid && result.models.length > 0) {
            const cleanModels = result.models.map((m) =>
              m.startsWith('models/') ? m.slice(7) : m
            );
            await db
              .update(appProviderKeys)
              .set({ models: cleanModels, updatedAt: new Date() })
              .where(eq(appProviderKeys.id, keyId));
          }
        })
        .catch(() => {}); // Silent failure -- non-blocking
    }
    ```

    Key details:
    - Returns `void` (not Promise) -- fire-and-forget, matches `recordAuditEvent()` and `logUsage()` patterns
    - Strips Google's `models/` prefix from model names (e.g., `models/gemini-1.5-pro` -> `gemini-1.5-pro`)
    - Only writes to DB if `result.valid === true` AND `result.models.length > 0`
    - Uses `.then().catch()` chain (no await at call site), same as `logUsage()` in `gateway/usage.ts`
    - The `db`, `appProviderKeys`, and `eq` imports are already at top of file

    Create `src/lib/server/provider-keys.test.ts` with vitest unit tests. Mock `validateProviderKey` (or the underlying fetch) and `db.update` to verify:

    2. Google prefix stripping works
    3. No DB write when models array is empty
    4. No throw/rejection when validateProviderKey fails

    Use `vi.mock` for the db module. The test should await the internal promise chain by testing the mocked db.update call with `vi.waitFor` or a small delay since discoverModelsForKey is fire-and-forget.
